# Atomic Plan — план по атомарным коммитам

Методичка: как превратить задачу в последовательность **атомарных коммитов**, где каждый коммит независимо собирается и проходит тесты. Это надстройка над планированием, а не замена: в native plan mode по-прежнему исследую код read-only и в конце вызываю `ExitPlanMode` на одобрение — скилл лишь задаёт структуру плана. Реализацию не начинать до одобрения.

## Инвариант коммита (green-to-green)

Каждый коммит из плана обязан:
1. **Собираться сам по себе** — модуль компилируется на этом коммите (прогон его тест-таска уже компилирует код; при сомнении — `:<module>:assemble`), без оглядки на следующие коммиты.
2. **Проходить тесты** — релевантные тесты зелёные (карта модуль→таск ниже). Дерево не краснеет даже временно.
3. **Быть одной логической правкой** — наименьшее осмысленное изменение, которое описывается одной строкой. Не «свалка из всего сразу».
4. **Нести свои тесты** — новые/изменённые тесты в том же коммите, что и код, который они покрывают. Никаких «тесты допишу следующим коммитом».

## Как резать

- **Порядок: аддитивное → переключение → удаление.** Сначала вводим новое (новые файлы/функции/перегрузки — ничего не ломают), затем переключаем вызовы на новое, удаление старого — последним коммитом. Так каждое промежуточное состояние зелёное.
- **Сквозная правка, которую нельзя сделать зелёной за один шаг** (напр. смена сигнатуры по многим файлам) — либо весь ripple одним коммитом (он всё ещё атомарен — одна логическая правка), либо временный shim/перегрузка, держащий старые вызовы живыми, и снос его финальным коммитом.
- **Не смешивать** рефактор и фичу в одном коммите. Чистый рефактор (поведение не меняется, тесты те же) — отдельным коммитом до или после.
- **Не тащить** в план попутные «улучшения», о которых не просили (CLAUDE.md → «Стиль работы пользователя»).

## Формат плана

План = пронумерованный список коммитов. На каждый:

> **Коммит N — `<заголовок в стиле текущего git log, часто «+ <область>: <суть>»>`**
> - **Что:** одна-две строки сути.
> - **Файлы:** какие трогаем.
> - **Тесты:** какие тесты добавляем/правим здесь же (стиль — skill `test-writer`).
> - **Проверка:** релевантный таск по карте ниже; `installDist` — если менялись CLI-флаги.
> - **Green:** собирается + тесты зелёные ✅

Карта проверки (из CLAUDE.md → «Команды и verification loop»):
- `SessionLoop`/`CliArgs`/`HistoryStore`/`MemoryStore`/`ContextStrategy`/`FactsExtractor`/`SessionStats` → `:cliJvmApp:test`.
- `*Api`/`HistoryCompressor`/`PricingRegistry`/`Profile`/`MemoryLayer` → `:shared:jvmTest`.
- Коммит затрагивает оба слоя → оба таска.
- Менялись CLI-флаги → плюс `:cliJvmApp:installDist` (иначе бинарь не знает новых флагов) и skill `cli-smoke`.

## Выполнение (после одобрения)

Коммит за коммитом, по одному. На каждом: правки → прогон проверки из плана → **только если зелёное** — коммит → следующий. Красный таск = стоп: не коммитить, чинить или пересматривать нарезку. Новые тесты — по skill `test-writer`. После коммитов, менявших CLI-флаги, — `cli-smoke`.

## Чего не делать

- **Красный промежуточный коммит** — даже «временно». Если шаг краснит дерево, нарезка неверная: укрупни коммит или добавь shim.
- **Коммит «только тесты, код был раньше»** или наоборот — тесты и код едут вместе.
- **Гигантский catch-all** «вся фича разом», когда её можно разбить green-to-green.
- **Размывать атомарность** ради красивого числа коммитов — один честный коммит лучше трёх, из которых два не собираются.
